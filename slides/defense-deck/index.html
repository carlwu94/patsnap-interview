TECH STACKWHY THESE CHOICES

技术选型

技术选型

Storage

SQLite + FTS5

单文件、零服务依赖、部署与演示成本低；同时支持结构化过滤和全文搜索，足够覆盖 500 篇专利的检索场景。

API

FastAPI

开发效率高，天然适合 REST + Streaming；本项目同时暴露查询接口与 NDJSON 聊天流，便于 Web 前端直接消费。

Tooling

FastMCP

把专利查询动作封装为可审计的 MCP Tool，既满足题目要求，也让 Agent 调用边界更清晰。

LLM

DeepSeek + OpenAI 兼容 SDK

保留 tool-calling 编程模型，同时适配本地可访问的 DeepSeek 服务，减少 provider 切换成本。

Parsing

PyMuPDF + RapidOCR

先走文本层提取，扫描件自动 fallback 到 OCR；最终命中 117 篇扫描类 PDF，全量 500/500 解析成功，0 失败。

Frontend

Jinja2 + Vanilla JS/CSS

演示页轻、依赖少、可控性强；流式输出、会话历史、Markdown 渲染都能直接按需求定制。

选型思路

这里不要逐个念库名，而是强调选型原则：第一是本地演示稳定，第二是工程复杂度不要超过题目规模，第三是每个选择都能解释清楚 tradeoff。

MCPTOOLS AS CAPABILITIES

MCP Tools

5 个 MCP Tools

设计原则

  • 一个 Tool 对应一个稳定意图：搜索、详情、趋势、IPC、公司名解析。
  • 参数尽量贴近业务语义，避免把 SQL/表结构泄漏到 Agent 层。
  • 先有 `list_assignees`，再做基于 `assignee` 的严格过滤，降低别名错误率。
  • Tool 结果可单独脚本验收，也可被 Agent 组合调用。

list_assignees

列出公司名称及专利数。作用是把“宁德时代”这类模糊别名收敛成数据库中的规范申请人名。

search_patents

支持关键词、申请人、年份组合检索；已支持公司+年份场景下 keyword 为空的合法查询。

get_patent_details

按专利号返回完整记录和抽取后的全文，用于问答时展开细节与原文依据。

get_top_ipc

输出专利集合中 IPC 主分类的高频分布，适合回答“哪些技术方向最集中”。

get_assignee_trend

针对单个公司返回年度申请趋势，适合回答“近几年布局变化”类问题。

验收方式

`scripts/demo_mcp_client.py` 会串行验证全部 5 个工具，保证 MCP 层可以独立证明可用。

能力封装

这一页建议重点讲 `list_assignees`。它不是额外功能，而是 Agent 稳定性的关键前置工具。没有它，后续 search 和 trend 的参数很容易带着别名或脏词走偏。

稳定性保护

  • 工具轮次上限从 4 提升到 8，但更关键的是重复工具调用检测。
  • 当检测到相同签名的工具调用反复出现时，强制“基于当前证据作答”。
  • 支持 history 透传，解决同一 session 多轮对话记忆缺失。
  • 当未配置模型时自动退回规则兜底，确保演示仍可完成。

前端可见效果

  • 聊天区可看到实时状态、工具调用与最终答复。
  • 会话历史在本地存储，支持切换不同 session。
  • 返回内容支持安全 Markdown 渲染，便于展示结构化答案。
一句话总结

Agent 负责“怎么问工具”，MCP Tool 负责“怎么查数据”，业务代码负责“怎么保证结果稳定”。

策略与边界

这里一定要强调：不要把稳定性寄托在 prompt 上。本项目真正解决问题的是代码侧的 history 传递、参数归一化和重复调用短路，这些都是工程控制，而不是玄学调 prompt。
